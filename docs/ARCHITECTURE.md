# Architecture

## Overview

The system is split into a Next.js frontend and an Express API backend. MongoDB stores profile, achievement, department, and document metadata. Files are uploaded to AWS S3 using short-lived signed URLs generated by the backend.

## Core flows

### Authentication

1. User logs in through `/api/auth/login`
2. Backend verifies credentials and returns a JWT
3. Frontend stores the token in local storage and includes it in authenticated API calls
4. Role-aware UI routes users to student or admin workspaces

### Student profile lifecycle

1. Student registers or is created by an admin
2. Student updates personal and academic profile fields
3. Student submits achievements and documents
4. Admin or faculty verifies achievement submissions
5. Analytics and reports aggregate approved and pending data for dashboard use

### Reporting

1. Admin dashboard loads KPI metrics and aggregations
2. Reports endpoint returns top achievers, department-level achievements, participation, and certification statistics
3. Export endpoint generates PDF or Excel reports on demand

## Security

- JWT authentication for all protected endpoints
- Role-based authorization middleware
- Zod request validation
- Helmet, rate limiting, CORS, and Mongo sanitize middleware
- Student access is scoped to self-owned data
- Faculty access is scoped to department-owned data

## Scalability notes

- Aggregation-heavy analytics are isolated in admin endpoints and can be cached later
- File traffic bypasses the API by using pre-signed uploads to cloud storage
- Frontend uses React Query to manage server-state caching and incremental revalidation
- Mongoose schemas include indexes on student ID, department, semester, CGPA, and achievement status/category
